Presentación de la Base de Datos Sign_bridge

Análisis del modelo, estructura de tablas y relaciones con diagrama gráfico en HTML.

Resumen general

La base de datos Sign_bridge está diseñada para una aplicación de traducción entre lenguaje de señas y texto/voz. Se trata de un modelo relacional MySQL orientado a sesiones de traducción, usuarios, multimedia y soporte.

El modelo incluye 15 tablas principales, con especial foco en:

  • Gestión de usuarios y permisos (Usuario, Rol, Region).
  • Gestión de sesiones de traducción (SesionTraduccion, EntradaVoz, EntradaSenas).
  • Soporte de vocabulario y traducciones (LexicalUni, TraduccionDetalle, PalabrasFavoritas).
  • Configuración personalizada (SignAvatarConfig, ConfiguracionDispositivo).
  • Auditoría y operaciones de soporte (Feedback, Soporte, LogAcceso, LogErrorSistema).

Estructura de tablas

Rol

Almacena los roles del sistema. Es tabla raíz sin dependencias y permite administrar permisos de usuarios.

Region

Define regiones geográficas de Colombia para asociar usuarios a ciudad y departamento.

Usuario

Tabla central. Relaciona usuario con rol y región, y es fuente principal para casi todas las demás tablas.

SesionTraduccion

Registra cada sesión de traducción y su estado. Es el eje transaccional del sistema.

EntradaVoz / EntradaSenas

Almacenan los datos multimedia de entrada asociados a sesiones de traducción.

LexicalUni

Catalogo de unidades léxicas (palabras/frases) para traducción y favoritos.

TraduccionDetalle

Relaciona sesiones con unidades léxicas, permitiendo reconstruir oraciones traducidas.

SignAvatarConfig / ConfiguracionDispositivo

Guardan preferencias visuales del avatar y configuración técnica de dispositivos, ambos en relación 1:1 con Usuario.

Feedback / Soporte / LogAcceso

Permiten medir satisfacción, registrar tickets y auditar accesos. Mantienen trazabilidad de la plataforma.

LogErrorSistema

Tabla independiente para errores críticos del sistema. No depende de usuarios, lo que la hace más robusta para fallas tempranas.

Patrones importantes

  • Se usan UUID como identificadores únicos en todas las tablas principales.
  • La eliminación lógica se habilita mediante deleted_at en la mayoría de tablas.
  • Existen relaciones 1:1 para configuración de avatar y dispositivo.
  • Feedback y soporte usan SET NULL en FKs para preservar histórico si el usuario o la sesión desaparece.
  • LogErrorSistema queda independiente para garantizar registro de errores incluso antes de inicio de sesión.

Estas inserciones ejemplifican la secuencia correcta: primero catálogos (Rol, Region, LexicalUni), luego Usuario, y finalmente entidades transaccionales.

Recomendaciones para la presentación

  • Muestra primero el propósito: traducir entre señas y voz/texto.
  • Explica el flujo principal: Usuario → Sesión → EntradaVoz/EntradaSenas → TraducciónDetalle.
  • Destaca la personalización y auditoría: avatar, dispositivo, feedback, soporte y logs.
  • Usa el diagrama ER para que sea más visual y fácil de entender para el público.
