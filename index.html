01 — Estructura

Tablas de la Base de Datos

Cada tabla fue diseñada siguiendo normalización y separación de responsabilidades. Las llaves primarias usan UUIDs para garantizar unicidad global.

PK — Llave Primaria
FK — Llave Foránea
Campos opcionales (NULL)

Rol

InnoDB
Catálogo central de roles del sistema. Define los niveles de acceso y permisos disponibles (admin, usuario, invitado, moderador, soporte). Facilita la gestión de permisos sin modificar la lógica de los usuarios.
id_rolCHAR(36)PK
nombre_rolVARCHAR(50)

Region

InnoDB
Catálogo de regiones geográficas de Colombia. Permite asociar usuarios a su ciudad y departamento para análisis geográficos y personalización regional del servicio LSC.
id_regionCHAR(36)PK
nombre_regionVARCHAR(100)
departamentoVARCHAR(100)
created_atDATETIME
updated_atDATETIME
deleted_atDATETIME NULL

Usuario

InnoDB
Núcleo de la plataforma. Almacena todos los usuarios registrados, los asocia a un rol y una región, garantiza unicidad del correo y almacena contraseñas como hash. De esta tabla dependen casi todas las demás entidades.
id_usuarioCHAR(36)PK
id_rolCHAR(36)FK
id_regionCHAR(36)FK
primer_nombreVARCHAR(50)
primer_apellidoVARCHAR(50)
telefonoVARCHAR(20)
correoVARCHAR(150) UNIQUE
password_hashVARCHAR(255)
deleted_atDATETIME NULL

SesionTraduccion

InnoDB
Eje transaccional del sistema. Registra cada sesión de traducción, gestiona el ciclo de vida (pendiente → procesando → completado/error) y determina la dirección (voz a señas o señas a texto).
id_sesionCHAR(36)PK
id_usuarioCHAR(36)FK
fecha_horaTIMESTAMP
estadoENUM(4 estados)
tipo_traduccionENUM(2 tipos)
deleted_atDATETIME NULL

EntradaVoz

InnoDB
Almacena el audio capturado y la transcripción generada por el motor STT (Speech-to-Text). Fundamental para el proceso de traducción voz → señas y auditoría de calidad.

id_entrada_senasCHAR(36)PK

InnoDB

InnoDB

03 — DML

Procedimientos de Actualización

Los UPDATE simulan operaciones reales del sistema: cambios de usuario, transiciones de estado del motor de traducción, correcciones editoriales y sincronización de configuraciones.

3.1 Renombrar Roles

UPDATE
UPDATE Rol SET nombre_rol = 'Administrador' WHERE nombre_rol = 'admin';
UPDATE Rol SET nombre_rol = 'Cliente'       WHERE nombre_rol = 'usuario';
¿Por qué? Mejora la semántica de los roles para mostrar nombres más legibles al usuario final en la interfaz, sin cambiar la lógica de negocio que ya usa los IDs como referencia.

3.2 Actualizar Datos de Usuarios

UPDATE
UPDATE Usuario SET primer_nombre  = 'Juan Actualizado' WHERE correo = 'juan@mail.com';
UPDATE Usuario SET password_hash = 'nuevo_hash'       WHERE correo = 'maria@mail.com';
¿Por qué? Simula el flujo de un usuario que cambia su nombre o restablece su contraseña. Se usa el correo como criterio porque tiene restricción UNIQUE, garantizando que solo se actualiza un registro.

3.3 Cambios de Estado en Sesiones

UPDATE
UPDATE SesionTraduccion SET estado = 'completado'   WHERE estado = 'pendiente';
UPDATE SesionTraduccion SET tipo_traduccion = 'senas_a_texto' WHERE tipo_traduccion = 'voz_a_senas';
¿Por qué? Simula la transición de estados en el ciclo de vida de una traducción. En producción, el motor de traducción actualizaría el estado automáticamente al finalizar el procesamiento.

3.4 Correcciones en Entradas de Voz

UPDATE
UPDATE EntradaVoz SET texto_generado = 'hola mundo'            WHERE url_audio = 'audio1.mp3';
UPDATE EntradaVoz SET url_audio      = 'audio_actualizado.mp3' WHERE url_audio = 'audio2.mp3';
¿Por qué? Representa la corrección manual de una transcripción errónea del STT o la actualización de la ruta del archivo de audio tras una reubicación en el servidor de almacenamiento.

3.5 Ajustes de Avatar (SignAvatarConfig)

UPDATE
UPDATE SignAvatarConfig SET ritmo_lsc  = 3          WHERE ritmo_lsc  = 1;
UPDATE SignAvatarConfig SET color_ropa = 'amarillo' WHERE color_ropa = 'rojo';
¿Por qué? Refleja la acción de un usuario que personaliza su avatar desde la interfaz. El campo ritmo_lsc controla el ritmo de animación en LSC. El sistema persiste estas preferencias entre sesiones.

3.6 Incremento de Contador de Palabras Favoritas

UPDATE
UPDATE PalabrasFavoritas SET veces_usado = veces_usado + 1 WHERE veces_usado = 5;
UPDATE PalabrasFavoritas SET veces_usado = 10                WHERE veces_usado = 2;
¿Por qué? Incremento atómico del contador de uso. Cada vez que el usuario usa una unidad léxica favorita en una traducción, el sistema actualiza este contador para alimentar el algoritmo de sugerencias de vocabulario.

3.7 Actualización de Estado de Tickets de Soporte

UPDATE
UPDATE Soporte SET estado = 'resuelto'   WHERE estado = 'pendiente';
UPDATE Soporte SET asunto = 'Actualizado' WHERE asunto = 'Bug';
¿Por qué? Simula el flujo de trabajo del equipo de soporte al cerrar tickets. En producción, este UPDATE sería ejecutado por el agente de soporte al marcar un ticket como resuelto.
04 — DML

Procedimientos de Borrado

Los DELETE se ejecutan en orden inverso a la creación de tablas (de tablas hijo a tablas padre), respetando las restricciones de integridad referencial. Cada sentencia usa WHERE con un criterio de negocio concreto para identificar los registros a eliminar.

⚠ Orden de Eliminación (tablas hijo → tablas padre)

EntradaVoz → EntradaSenas → TraduccionDetalle → Feedback → Soporte → PalabrasFavoritas → LogAcceso → SesionTraduccion → SignAvatarConfig → ConfigDispositivo → Usuario → Rol · LexicalUni · LogErrorSistema

4.1 Eliminar entradas de voz sin audio cargado

DELETE
-- Entradas de voz cuyo archivo de audio no fue cargado (NULL)
DELETE FROM EntradaVoz
WHERE url_audio IS NULL;

-- Entradas de voz sin transcripción generada por el motor STT
DELETE FROM EntradaVoz
WHERE texto_generado IS NULL;
¿Por qué? Elimina entradas de voz incompletas que no tienen archivo de audio ni transcripción: registros que no aportan valor al sistema y ocupan espacio innecesario. Criterio de negocio: una entrada sin audio o sin texto generado no puede ser traducida.

4.2 Eliminar entradas de señas sin video

DELETE
-- Entradas de señas sin video adjunto
DELETE FROM EntradaSenas
WHERE url_video IS NULL;

-- Entradas de señas sin texto reconocido por la IA
DELETE FROM EntradaSenas
WHERE texto_generado IS NULL;
¿Por qué? Depura entradas de señas vacías que el modelo de IA no pudo procesar. Sin video ni texto reconocido, estos registros son inútiles para el entrenamiento del modelo y para la auditoría de calidad.

4.3 Eliminar detalles de traducción con orden inválido

DELETE
-- Detalles con posición 0 (orden inválido en la oración traducida)
DELETE FROM TraduccionDetalle
WHERE orden = 0;

-- Detalles con posición negativa (error de inserción)
DELETE FROM TraduccionDetalle
WHERE orden < 0;
¿Por qué? El campo orden define la posición de cada unidad léxica en la oración. Un valor de 0 o negativo indica un error de inserción: el sistema usa posiciones a partir de 1, por lo que estos registros romperían la reconstrucción correcta de la oración traducida.

4.4 Eliminar feedback con calificación fuera de rango

DELETE
-- Feedback con calificación menor a 1 (datos corruptos)
DELETE FROM Feedback
WHERE calificacion < 1;

-- Feedback con calificación mayor a 5 (fuera del rango permitido)
DELETE FROM Feedback
WHERE calificacion > 5;
¿Por qué? La escala de calificación válida es de 1 a 5. Registros con valores fuera de ese rango son datos corruptos o errores de ingreso que distorsionarían el promedio de satisfacción y los reportes de analítica del sistema.

4.5 Eliminar tickets de soporte ya resueltos sin asunto

DELETE
-- Tickets resueltos sin asunto registrado (incompletos)
DELETE FROM Soporte
WHERE estado = 'resuelto'
  AND asunto IS NULL;

-- Tickets resueltos sin mensaje del usuario
DELETE FROM Soporte
WHERE estado = 'resuelto'
  AND mensaje IS NULL;
¿Por qué? Los tickets resueltos sin asunto ni mensaje son registros vacíos que no aportan información histórica al equipo de soporte. Solo se eliminan los ya resueltos para no afectar tickets activos que puedan estar en proceso de llenado.

4.6 Eliminar palabras favoritas sin uso

DELETE
-- Favoritos que nunca fueron utilizados por el usuario
DELETE FROM PalabrasFavoritas
WHERE veces_usado = 0;

-- Favoritos marcados como eliminados lógicamente
DELETE FROM PalabrasFavoritas
WHERE deleted_at IS NOT NULL;
¿Por qué? Palabras favoritas con veces_usado = 0 fueron marcadas pero nunca usadas, lo cual distorsiona el algoritmo de sugerencias. Los registros con deleted_at definido ya estaban marcados para eliminación lógica y se purgan definitivamente.

4.7 Eliminar logs de accesos fallidos

DELETE
-- Logs de intentos fallidos de acceso (limpieza periódica)
DELETE FROM LogAcceso
WHERE tipo_acceso = 'intento_fallido';

-- Logs marcados como eliminados lógicamente
DELETE FROM LogAcceso
WHERE deleted_at IS NOT NULL;
¿Por qué? Los intentos fallidos acumulados pueden ser depurados periódicamente tras su análisis de seguridad. Mantenerlos indefinidamente hincha la tabla sin beneficio operativo. Los registros con deleted_at ya fueron auditados y pueden ser purgar del sistema.

4.8 Eliminar sesiones de traducción con error

DELETE
-- Sesiones que terminaron en error y ya fueron revisadas
DELETE FROM SesionTraduccion
WHERE estado = 'error';

-- Sesiones marcadas como eliminadas lógicamente
DELETE FROM SesionTraduccion
WHERE deleted_at IS NOT NULL;
¿Por qué? Las sesiones con estado 'error' son traducciones fallidas que no generaron resultado útil para el usuario. Después de ser analizadas por el equipo técnico, pueden eliminarse para liberar espacio y mantener limpia la tabla transaccional.

4.9 Eliminar configuraciones de avatar y dispositivo sin usar

DELETE
-- Avatares con estilo nulo (nunca configurados)
DELETE FROM SignAvatarConfig
WHERE estilo_avatar IS NULL;

-- Configuraciones de dispositivo sin versión de Android registrada
DELETE FROM ConfiguracionDispositivo
WHERE version_android IS NULL;
¿Por qué? Configuraciones de avatar sin estilo y dispositivos sin versión de Android son registros generados automáticamente pero nunca completados por el usuario. Son filas vacías que ocupan espacio sin ofrecer valor funcional al sistema.

4.10 Eliminar usuarios inactivos y roles obsoletos

DELETE
-- Usuarios marcados como eliminados lógicamente (soft delete)
DELETE FROM Usuario
WHERE deleted_at IS NOT NULL;

-- Roles que ya no existen en el sistema
DELETE FROM Rol
WHERE nombre_rol = 'invitado';

-- Vocabulario marcado como eliminado lógicamente
DELETE FROM LexicalUni
WHERE deleted_at IS NOT NULL;

-- Errores del sistema ya resueltos (módulo dado de baja)
DELETE FROM LogErrorSistema
WHERE modulo = 'legacy_stt';
¿Por qué? Fase final de limpieza: se purgan usuarios previamente marcados con soft delete, el rol 'invitado' que fue retirado del sistema, vocabulario obsoleto y errores de módulos ya desactivados. Siempre se eliminan las tablas raíz al final para no romper la integridad referencial.
05 — DML Avanzado

Operaciones MERGE (Upsert)

El patrón INSERT ... ON DUPLICATE KEY UPDATE permite insertar un registro si no existe, o actualizarlo si ya existe. Ideal para sincronización de datos sin generar duplicados.

5.1 Merge de Usuario

MERGE
INSERT INTO Usuario (id_usuario, id_rol, primer_nombre, primer_apellido, telefono, correo, password_hash)
SELECT id_usuario, id_rol, primer_nombre, primer_apellido, telefono, correo, password_hash
FROM Usuario AS nuevo
ON DUPLICATE KEY UPDATE
  primer_nombre  = nuevo.primer_nombre,
  primer_apellido= nuevo.primer_apellido,
  telefono       = nuevo.telefono,
  password_hash  = nuevo.password_hash,
  id_rol         = nuevo.id_rol;
¿Por qué? Permite sincronizar perfiles de usuario desde un sistema externo (ej. SSO corporativo) sin duplicar registros. Si el usuario ya existe, solo se actualizan los campos relevantes; si no existe, se crea.

5.2 Merge de LexicalUni

MERGE
INSERT INTO LexicalUni (id_LexicalUni, texto, idioma)
SELECT id_LexicalUni, texto, idioma FROM LexicalUni AS nueva
ON DUPLICATE KEY UPDATE
  texto  = nueva.texto,
  idioma = nueva.idioma;
¿Por qué? Facilita la carga incremental del diccionario de señas desde fuentes externas (ej. corpus académicos de LSC) sin eliminar las entradas existentes.

5.3 Merge de SignAvatarConfig

MERGE
INSERT INTO SignAvatarConfig
  (id_sing_avatar, id_usuario, estilo_avatar, color_piel, color_ropa, tamano_avatar, ritmo_lsc)
SELECT id_sing_avatar, id_usuario, estilo_avatar, color_piel, color_ropa, tamano_avatar, ritmo_lsc
FROM SignAvatarConfig AS nuevo_av
ON DUPLICATE KEY UPDATE
  estilo_avatar = nuevo_av.estilo_avatar,
  color_piel    = nuevo_av.color_piel,
  color_ropa    = nuevo_av.color_ropa,
  tamano_avatar = nuevo_av.tamano_avatar,
  ritmo_lsc     = nuevo_av.ritmo_lsc;
¿Por qué? Sincroniza la configuración del avatar desde la app móvil al servidor central. Si el usuario ya tiene configuración guardada, se actualiza; si no, se crea por primera vez.
06 — Consultas

10 Consultas SELECT con Operadores Avanzados

Cada consulta utiliza operadores avanzados de SQL (LIKE, BETWEEN, IN, IS NULL) para extraer información relevante del sistema SignBridge.

01

Usuarios por dominio de correo

LIKE 
SELECT * FROM Usuario
WHERE correo LIKE '%@mail.com';
Relevancia: Segmenta usuarios por proveedor de correo electrónico. Útil para campañas de marketing dirigidas a un dominio corporativo específico, o migración de cuentas cuando un proveedor cierra. El comodín % antes del @ captura cualquier nombre de usuario.
02

Vocabulario en español (con prefijo)

LIKE 
SELECT * FROM LexicalUni
WHERE idioma LIKE 'es%';
Relevancia: Filtra todas las unidades léxicas en variantes del español (es, es_Co, es_MX, es_AR) usando el prefijo 'es'. Fundamental para el motor de traducción cuando necesita cargar solo el vocabulario del idioma activo del usuario.
03

Feedback positivo (calificaciones altas)

BETWEEN 
SELECT * FROM Feedback
WHERE calificacion BETWEEN 4 AND 5;
Relevancia: Identifica usuarios altamente satisfechos. Estos registros se usan para solicitar testimonios, identificar funcionalidades exitosas o seleccionar usuarios para beta testing. BETWEEN es más legible que >= 4 AND <= 5.
04

Feedback negativo ordenado por gravedad

BETWEEN + ORDER BY 
SELECT * FROM Feedback
WHERE calificacion BETWEEN 1 AND 2
ORDER BY calificacion ASC;
Relevancia: Detecta las peores evaluaciones del servicio, ordenadas de más grave a menos grave. Permite al equipo de producto priorizar la revisión de comentarios negativos y tomar acciones correctivas con mayor impacto.
05

Sesiones con problemas (no completadas)

IN 
SELECT * FROM SesionTraduccion
WHERE estado IN ('pendiente', 'error');
Relevancia: Identifica todas las sesiones que no han sido completadas exitosamente. Permite al equipo técnico monitorear procesos atascados y al sistema enviar notificaciones o reintentar el procesamiento. IN es más eficiente y legible que múltiples condiciones OR.
06

Auditoría de accesos legítimos

IN + ORDER BY DESC 
SELECT * FROM LogAcceso
WHERE tipo_acceso IN ('login', 'logout')
ORDER BY fecha_hora DESC;
Relevancia: Extrae los accesos legítimos ordenados del más reciente al más antiguo. Permite al equipo de seguridad auditar actividad y detectar sesiones muy largas (login sin logout correspondiente). El DESC muestra primero los eventos más recientes.
07

Feedback sin usuario asociado (huérfanos)

IS NULL 
SELECT * FROM Feedback
WHERE id_usuario IS NULL;
Relevancia: Detecta registros de feedback huérfanos cuyo usuario fue eliminado y quedó en NULL por ON DELETE SET NULL. Permite identificar inconsistencias o programar tareas de limpieza periódica. IS NULL es obligatorio; el operador = NULL no funciona en SQL.
08

Tickets de soporte activos

OR 
SELECT * FROM Soporte
WHERE estado = 'pendiente'
   OR estado = 'en_proceso';
Relevancia: Obtiene todos los tickets que aún requieren atención. En un panel de administración, esta consulta alimentaría la cola de trabajo del agente de soporte, mostrando solo los casos abiertos y excluyendo los ya resueltos.
09

Dispositivos offline con Android reciente

AND + LIKE 
SELECT * FROM ConfiguracionDispositivo
WHERE uso_offline = TRUE
  AND version_android LIKE 'Android 1%';
Relevancia: Identifica usuarios con modo offline activado en versiones recientes de Android (10 en adelante). Permite al equipo de desarrollo priorizar pruebas de compatibilidad offline en los dispositivos más utilizados por este segmento.
10

Palabras favoritas más frecuentes

> + ORDER BY DESC 
SELECT * FROM PalabrasFavoritas
WHERE veces_usado > 2
ORDER BY veces_usado DESC;
Relevancia: Identifica el vocabulario más utilizado por los usuarios. Estos datos alimentan el algoritmo de predicción de palabras y la personalización de sugerencias en la interfaz de traducción. Ordenado de mayor a menor frecuencia para priorizar las palabras con mayor impacto.
